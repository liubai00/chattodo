# AI Todo v0.2 Iteration Requirements

版本：v0.2 draft  
用途：第一版 demo 后的迭代需求文档  
读者：产品、设计、研发、内部测试成员  

---

## 1. 背景

第一版 demo 已经帮助团队验证了 AI Todo 的基本方向：用户可以通过自然语言输入，让 AI 判断内容是否应该进入 todo 系统。

下一阶段的目标不是做完整商业化产品，而是让内部团队可以真实试用。

v0.2 需要从“能演示”走向“能每天用”：

- 用户要能登录。
- Web 要成为主要入口。
- AI 要接入真实能力。
- 任务要能被用户校对和编辑。
- 团队要能看到内部测试数据。
- AI 的生成结果要能追溯到原始输入。

---

## 2. 本轮目标

v0.2 的核心目标：

> 让内部团队可以在网页版中真实使用 AI Todo，完成从聊天输入、AI 生成、任务校对、数据库管理到后台观察的闭环。

本轮不追求功能完整，而是优先保证：

- 可用：内部用户能完成核心流程。
- 可观察：团队能看到真实测试数据。
- 可纠错：AI 判断错了，用户能快速修正。
- 可追溯：任务能回到原始输入和 AI 生成记录。

---

## 3. 本轮范围

### 3.1 账号系统

需要支持内部测试用户登录。

产品要求：

- 用户可以登录进入自己的工作区。
- 用户数据相互隔离。
- 管理员可以进入内部后台查看所有测试数据。

不要求：

- 商业化注册流程。
- 复杂组织权限。
- 计费、套餐、额度。

### 3.2 Web 优先

v0.2 暂停 CLI，优先完善网页版。

Web 需要成为主要使用入口，支持：

- 聊天输入。
- Todo 数据库管理。
- 任务详情查看。
- 待澄清内容处理。
- 非 todo 内容查看。
- Agent 和设置入口。

### 3.3 三段式响应式布局

桌面端采用三段式布局：

```text
左侧图标功能栏 / 中间列表或对象区 / 右侧聊天与详情区
```

参考产品：

- 微信
- ChatGPT
- 飞书

左侧功能栏第一版包含：

- 聊天
- Todo 数据库
- Agent
- 设置

移动端不强制三栏同时展示，但必须能完成核心流程。

### 3.4 聊天生成任务

聊天是 v0.2 的主流程。

用户输入自然语言后，Agent 先判断内容类型：

- 明确 todo：自动进入 Todo 数据库。
- 模糊 todo：进入轻量待澄清区。
- 非 todo：进入隔离区。

明确 todo 自动入库，不需要用户每次确认。  
如果 AI 分类错误，用户通过“移出 todo”修正。

### 3.5 Todo 数据库

Todo 数据库参考 Notion 或多维表格体验。

用户需要能：

- 查看正式任务。
- 手动编辑任务核心字段。
- 打开任务详情。
- 查看来源原文。
- 查看 AI 生成记录。
- 将误分类任务移出 todo。

建议第一版可编辑字段：

- 状态
- 标题
- 项目
- 标签
- 截止时间
- 优先级
- 备注

字段和交互细节由技术同学结合现有 demo 实现决定。

### 3.6 待澄清区

保留待澄清区，但弱化它的主流程地位。

进入待澄清区的内容是：

- 有行动倾向。
- 但目标、下一步或完成标准不清晰。

用户可以后续补充信息，再转为正式任务。

### 3.7 非 todo 隔离区

非 todo 内容不进入任务系统。

它可以是：

- 灵感。
- 观点。
- 参考信息。
- 摘录。
- 不需要行动的备忘。

非 todo 不参与：

- 今日任务。
- 下一步计划。
- 任务统计。

用户可以手动转成 todo，但必须明确触发。

### 3.8 误分类移出

Todo 数据库中的任务需要提供“移出 todo”能力。

使用场景：

AI 把一条非 todo 信息错误识别成任务，用户检查数据库时发现问题。

用户操作后：

- 该条目不再作为任务存在。
- 进入非 todo 或隔离状态。
- 保留来源原文和 AI 生成记录。
- 系统记录这是一次用户纠错。

### 3.9 Raw Data 与 AI 生成记录

每次用户输入都要保存为 raw data。

AI 基于 raw data 生成结果时，需要有生成记录。

用户在任务详情中至少能看到：

- 原始输入。
- AI 判断结果。
- 生成时间。
- 当前任务内容。

如果任务被用户修改或移出，需要能保留基本记录。

本轮不要求复杂审计系统，但需要能支撑用户校对和团队排查问题。

### 3.10 `@` 引用

聊天框支持 `@` 引用。

v0.2 第一版只支持：

- `@任务`
- `@项目`

用户输入 `@` 后，可以搜索并选择已有任务或项目，让 AI 基于引用对象继续对话。

暂不支持：

- `@Agent`
- `@非 todo`
- `@待澄清条目`
- `@任意数据库记录`

### 3.11 真实 AI API 接入

v0.2 需要从 mock 或规则判断，进入真实 AI API 接入阶段。

产品要求：

- AI 可以判断用户输入类型。
- AI 可以生成任务或待澄清内容。
- AI 可以识别非 todo。
- AI 失败时，用户能看到状态。
- 团队能在后台看到失败或异常记录。

具体模型、供应商、成本控制、重试策略由技术同学决定。

### 3.12 Codex 登录预研

产品希望支持真实 Codex 账号登录或授权。

但该能力存在技术不确定性，因此 v0.2 将它定义为预研项：

- 确认官方支持哪些认证方式。
- 判断是否能在我们的产品中接入真实 Codex 登录。
- 如果不能，给出替代方案。

可能替代方向包括：

- OpenAI API key。
- OpenAI access token。
- 用户本地 Codex 登录状态。
- 管理员统一配置。

参考官方资料：

- [Codex Authentication](https://developers.openai.com/codex/auth)
- [Codex CLI reference](https://developers.openai.com/codex/cli/reference)
- [Codex Access Tokens](https://developers.openai.com/codex/enterprise/access-tokens)

验收结果不要求一定打通完整 Codex 登录，但要求技术同学给出清晰结论。

### 3.13 内部只读后台

新增内部后台，方便团队观察测试数据。

后台第一版只读，不做数据编辑。

需要能查看：

- 用户列表。
- 用户会话。
- 原始输入。
- AI 生成记录。
- Todo 数据。
- 待澄清数据。
- 非 todo 数据。
- 错误或失败状态。

后台的目标是帮助团队发现问题，不是做正式运营系统。

## 4. 非本轮范围

v0.2 不做：

- CLI。
- 移动原生 App。
- 商业化账号体系。
- 复杂组织权限。
- 计费。
- 多 Agent 编排。
- 飞书、微信、Obsidian、Cubox 等外部输入集成。
- 完整知识库。
- 完整端到端加密。
- 完整审计系统。
- 复杂通知系统。

## 5. 核心验收场景

### 5.1 内部用户登录

内部用户可以登录网页版，并进入自己的工作区。

验收：

- 用户 A 看不到用户 B 的个人任务数据。
- 管理员可以在后台只读查看用户 A 和用户 B 的测试数据。

### 5.2 明确 todo 自动入库

用户在聊天框输入：

```text
下周三前把 AI todo app 的 v0.2 迭代文档发给团队
```

验收：

- 系统保存原始输入。
- AI 判断为明确 todo。
- Todo 数据库出现对应任务。
- 用户可以打开任务详情查看来源原文和生成记录。

### 5.3 模糊 todo 进入待澄清区

用户输入：

```text
周末研究一下几个任务管理工具
```

验收：

- 系统保存原始输入。
- AI 判断为模糊 todo。
- 内容进入待澄清区。
- 用户可以后续补充信息并转成正式任务。

### 5.4 非 todo 进入隔离区

用户输入：

```text
这个产品的体验可以参考飞书的三栏布局
```

验收：

- 系统保存原始输入。
- AI 判断为非 todo。
- 内容进入隔离区。
- 该内容不出现在 Todo 数据库。
- 该内容不参与下一步计划。

### 5.5 用户手动编辑任务

用户在 Todo 数据库中编辑任务。

验收：

- 可以修改任务标题、状态、项目、标签、截止时间、优先级或备注。
- 修改后数据保存成功。
- 刷新页面后仍能看到修改结果。

### 5.6 移出误分类 todo

用户发现 Todo 数据库中有一条不应该做的内容。

验收：

- 用户可以点击“移出 todo”。
- 该条目从 Todo 数据库移出。
- 该内容进入隔离状态或非 todo 区。
- 原始输入和生成记录仍然可追溯。

### 5.7 查看 raw data 与生成记录

用户打开一个任务详情。

验收：

- 能看到该任务来自哪条原始输入。
- 能看到 AI 的判断类型。
- 能看到生成时间。
- 如果用户修改过任务，至少能看出当前内容和来源之间的关系。

### 5.8 `@` 引用任务或项目

用户在聊天框输入 `@`。

验收：

- 可以搜索任务或项目。
- 选择引用对象后，对话中显示引用关系。
- AI 能基于被引用对象继续回答或生成任务。

### 5.9 内部后台查看数据

管理员进入内部后台。

验收：

- 可以查看所有内部测试用户。
- 可以查看用户的 raw input、任务、待澄清内容、非 todo 内容和生成记录。
- 后台只读，不能误改用户数据。

### 5.10 响应式 Web

用户在桌面和移动端访问 Web。

验收：

- 桌面端能使用三段式布局。
- 移动端能完成聊天输入、查看生成结果、编辑任务、移出误分类内容。
- 页面不出现明显内容溢出或遮挡。

### 5.11 Codex 登录预研结论

技术同学完成 Codex 登录可行性判断。

验收：

- 明确是否能接入真实 Codex 登录。
- 明确推荐实现路径。
- 明确不可行时的替代方案。
- 明确后续需要的账号、权限或安全边界。

## 6. 建议优先级

### P0

- 内部账号登录。
- Web 主布局。
- 聊天输入。
- 明确 todo 自动入库。
- Todo 数据库查看和编辑。
- raw data 与生成记录。
- 误分类移出。
- 真实 AI API 接入。

### P1

- 待澄清区。
- 非 todo 隔离区。
- `@` 引用任务和项目。
- 内部只读后台。
- 响应式优化。

### P2

- Agent 设置优化。
- Codex 登录打通或替代方案落地。
- 更丰富的筛选和视图。
- 数据导出。

优先级可由技术同学结合现有 demo 状态调整，但调整后需要保证内部真实试用闭环不被破坏。

## 8. 产品验收标准

产品侧验收不检查具体技术实现，只检查用户流程是否成立。

v0.2 可以被认为完成，当内部测试用户能够连续完成以下流程：

1. 登录 Web。
2. 在聊天框输入真实工作任务。
3. 看到 AI 自动生成任务。
4. 在 Todo 数据库中编辑任务。
5. 打开任务查看来源原文和生成记录。
6. 将错误分类的任务移出。
7. 使用 `@` 引用已有任务或项目继续对话。
8. 管理员在后台看到这次测试产生的数据。

## 9. 风险与待确认

### 9.1 AI 分类不稳定

第一阶段允许 AI 判断不完美，但必须提供纠错和追溯。

### 9.2 Codex 登录不确定

真实 Codex 登录可能不具备第三方产品可直接接入的标准能力。

需要技术同学预研官方能力，并给出实现路径或替代方案。

### 9.3 后台暴露数据的边界

当前是内部测试，所以后台可以查看所有测试数据。  
如果后续面向外部用户，后台权限、隐私和审计需要重新设计。

### 9.4 Web 布局复杂度

三段式布局适合桌面端，但移动端需要折叠和切换。  
v0.2 以可用为主，不追求移动端体验完整。

## 10. 给研发的说明

本文档只定义产品目标和验收行为，不替代技术设计。

以下内容由研发判断：

- 技术栈。
- 数据表设计。
- API 设计。
- 模型供应商。
- 鉴权方案。
- 后台实现方式。
- 部署方式。
- 日志和监控方案。

产品侧只要求最终用户流程、后台观察能力和数据追溯能力成立。
