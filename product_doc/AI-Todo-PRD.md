# AI Todo Product Requirements Document

版本：v0.2 draft  
用途：作为 AI Todo 的长期产品基线文档  
读者：产品、设计、研发、内部测试成员  

---

## 1. 产品定位

AI Todo 是一个 **AI 原生的 todo 类想法收集与处理系统**。

用户可以把想法、任务、备忘、对话片段、参考信息丢给系统；系统的核心职责不是保存所有信息，而是判断哪些内容需要行动，并把这些内容转化为可执行、可校对、可推进的 todo。

一句话：

> AI Todo 帮用户把自然语言里的行动意图，变成可管理、可追踪、可推进的任务系统。

---

## 2. 产品边界

### 2.1 Todo first

AI Todo 的主系统只服务 todo。

一个输入进入系统后，AI 需要先判断它属于哪一类：

1. 明确 todo：可以直接进入 Todo 数据库。
2. 模糊 todo：有行动倾向，但还需要澄清，进入待澄清区。
3. 非 todo 信息：进入隔离区，不参与任务计划。

这意味着：

- 产品不是通用知识库。
- 产品不是稍后读工具。
- 产品不是普通聊天机器人。
- 产品不是文档收藏系统。

信息可以被保存，但它的价值是帮助用户决定：

- 要不要做？
- 下一步是什么？
- 什么时候做？
- 谁负责？
- 做到什么程度算完成？

### 2.2 AI 是主交互，不是附加功能

AI Todo 的主入口是聊天。

用户不需要先理解复杂表单，也不需要一开始就把任务拆得很清楚。用户可以先自然语言输入，再由 AI 进行判断、拆解、归类和追问。

同时，AI 生成的结果必须允许用户校对和修正。AI 负责提高效率，但不能剥夺用户对任务系统的控制权。

### 2.3 可纠错比一次判断正确更重要

第一阶段不追求 AI 每次都完全正确，而是要让错误容易被发现、容易被修正。

因此，系统需要保留：

- 用户原始输入。
- AI 判断结果。
- AI 生成的任务或条目。
- 用户后续的校对和移出操作。

---

## 3. 目标用户

第一阶段默认服务内部团队和高频知识工作者。

典型用户特征：

- 每天产生很多零散想法、任务和提醒。
- 工作和个人事务容易混在一起。
- 习惯使用 AI、聊天工具、文档工具或任务管理工具。
- 希望降低整理任务的心理成本。
- 希望 AI 帮自己把输入转成可执行行动，而不是堆积更多信息。

---

## 4. 核心痛点

### 4.1 输入成本高

传统 todo 工具要求用户在输入时就填标题、项目、截止时间、标签、优先级等字段。

真实场景里，用户经常只有一句模糊想法：

```text
周末研究一下 Cubox、OmniFocus、Todoist
```

这类输入如果要求用户马上结构化，很容易被放弃。

### 4.2 信息和行动混在一起

用户会同时记录任务、灵感、摘录、参考信息和想法。如果这些内容都进入同一个列表，任务系统会越来越脏，最后很难判断什么真的需要做。

### 4.3 AI 生成结果不可追溯

如果 AI 直接生成任务，但用户看不到原始输入和判断过程，就很难校对，也很难信任系统。

### 4.4 现有任务系统缺少自然语言上下文

很多任务列表只保存标题和字段，但用户后续想问：

```text
这个任务当时是从哪句话来的？
```

系统需要能把任务和原始输入关联起来。

## 5. 核心产品流程

### 5.1 聊天输入

用户在聊天框中输入自然语言。

示例：

```text
下周三前把 AI todo app 的 MVP 文档提交评审
```

系统保存这条原始输入，并交给 Agent 判断。

### 5.2 Agent 判断

Agent 判断输入属于三类之一：

- 明确 todo
- 模糊 todo
- 非 todo

判断结果需要能被用户看到或追溯，不要求在主界面强展示全部技术细节。

### 5.3 明确 todo 自动入库

如果输入是明确 todo，系统自动生成任务并进入 Todo 数据库。

任务可以包含：

- 标题
- 状态
- 项目
- 标签
- 截止时间
- 优先级
- 备注
- 来源原文
- AI 生成记录

字段细节由技术同学结合实现方案确认，PRD 只要求用户能编辑核心字段。

### 5.4 模糊 todo 进入待澄清区

如果输入有行动倾向，但缺少目标、完成标准或下一步，则进入待澄清区。

待澄清区是轻量辅助区，不是 v0.2 的主工作区。它的作用是提醒用户：这里有一些可能要做，但还没有变成正式任务的内容。

### 5.5 非 todo 进入隔离区

如果输入只是观点、灵感、参考、摘录或非行动信息，则进入非 todo 隔离区。

非 todo 默认不参与：

- 下一步计划
- 今日任务
- 任务统计
- 执行提醒

用户后续可以手动把非 todo 转成 todo，但必须由用户明确触发。

### 5.6 用户纠错

如果 AI 把非 todo 错误放进 Todo 数据库，用户可以在任务条目上点击“移出 todo”。

移出后：

- 该内容不再参与任务计划。
- 原始输入和生成记录仍然保留。
- 系统记录这是一次用户纠错。

## 6. Web 信息架构

v0.2 起，产品优先完善网页版，CLI 暂缓。

### 6.1 总体布局

桌面端采用三段式布局，参考微信、ChatGPT、飞书等产品的工作区结构：

```text
左侧：图标功能栏
中间：列表 / 对象导航区
右侧：聊天与详情工作区
```

### 6.2 左侧功能栏

左侧只展示图标和必要状态，承担全局导航。

第一阶段包含：

- 聊天
- Todo 数据库
- Agent
- 设置

后续可加入：

- 项目
- 待澄清区
- 非 todo 隔离区
- 内部后台

### 6.3 中间对象区

中间区域根据当前功能变化。

在聊天模块中，它展示会话或联系人列表。  
在 Todo 数据库中，它展示视图、项目或筛选条件。  
在 Agent 模块中，它展示 Agent 列表或 Agent 配置入口。

### 6.4 右侧工作区

右侧是主要操作区域。

它可以是：

- 聊天窗口
- 任务详情
- Todo 数据库表格
- Agent 设置
- 后台数据详情

### 6.5 响应式要求

Web 必须在桌面和移动端都可用。

桌面端优先保证三段式效率。  
移动端可以折叠为分层切换，不要求保持三栏同时显示。

移动端最低要求：

- 能输入聊天。
- 能查看生成结果。
- 能编辑任务。
- 能移出错误分类内容。

## 7. 核心模块

### 7.1 Chat

Chat 是产品主入口。

用户可以：

- 输入新想法。
- 询问下一步计划。
- 围绕已有任务继续对话。
- 使用 `@` 引用任务或项目。

v0.2 的 `@` 只支持任务和项目，避免第一版搜索范围过大。

### 7.2 Todo 数据库

Todo 数据库是正式任务的主视图，参考 Notion 或多维表格的使用体验。

用户可以：

- 查看所有正式 todo。
- 手动编辑核心字段。
- 筛选不同状态或项目。
- 打开任务详情。
- 查看任务来源原文。
- 查看 AI 生成记录。
- 将误分类任务移出 todo。

### 7.3 待澄清区

待澄清区存放模糊 todo。

用户可以：

- 查看待澄清条目。
- 补充信息。
- 转为正式任务。
- 放弃或移出。

v0.2 中，待澄清区保留但弱化，不作为主导航中的强入口。

### 7.4 非 todo 隔离区

非 todo 隔离区存放不进入任务系统的信息。

用户可以：

- 查看被 AI 判断为非 todo 的内容。
- 复制或导出。
- 手动转成 todo。
- 归档或删除。

非 todo 不参与任务计划，除非用户明确引用或转换。

### 7.5 Agent

Agent 负责理解用户输入并生成结构化结果。

用户需要能配置：

- Agent 的基本风格。
- 偏好。
- 记忆或背景信息。
- 默认判断原则。

具体 prompt、模型和服务实现由技术同学设计。

### 7.6 设置

设置包含：

- 账号信息。
- AI 接入配置。
- Codex 登录或授权状态。
- 默认工作区。
- 隐私或可见范围。

### 7.7 内部后台

内部后台服务团队测试，不面向外部用户。

第一阶段后台只读，方便团队观察：

- 用户数据。
- 会话数据。
- 原始输入。
- 任务数据。
- AI 生成记录。
- 错误和失败状态。

内部后台的目标是帮助团队发现问题，而不是做运营系统。

---

## 8. 账号与权限

v0.2 需要账号系统，让内部测试用户能登录并隔离自己的数据。

第一阶段只要求支持内部测试闭环：

- 用户能登录。
- 用户只能看到自己的数据。
- 管理员能在内部后台只读查看所有测试数据。

不要求第一阶段完成：

- 商业化组织体系。
- 复杂角色权限。
- 计费。
- 多租户企业管理。

---

## 9. AI 接入与 Codex 登录

### 9.1 AI API 接入

v0.2 需要接入真实 AI API，替代纯 mock 或规则判断。

产品侧要求：

- 用户输入后，AI 能完成基础分类。
- AI 能生成任务、待澄清项或非 todo 输出。
- AI 生成过程有记录可追溯。
- AI 失败时，用户能看到明确状态，而不是静默失败。

模型、供应商、调用方式、成本控制由技术同学评估。

### 9.2 Codex 登录诉求

产品希望探索真实 Codex 账号登录能力，让用户可以在产品中连接自己的 Codex/OpenAI 相关身份或能力。

这部分是 v0.2 的技术风险项，需要研发预研：

- 官方是否支持第三方产品内嵌式 Codex 登录。
- 可行的授权方式是什么。
- 是否只能支持 API key、access token 或本地 Codex 登录状态。
- 如果真实 Codex 登录不可行，替代方案是什么。

参考官方资料：

- [Codex Authentication](https://developers.openai.com/codex/auth)
- [Codex CLI reference](https://developers.openai.com/codex/cli/reference)
- [Codex Access Tokens](https://developers.openai.com/codex/enterprise/access-tokens)

PRD 不把 Codex 登录写成确定可交付功能，而是写成明确产品诉求和技术预研事项。

## 10. Raw Data 与生成记录

每次用户输入都应先保存为 raw data。

AI 基于 raw data 生成任务、待澄清项或非 todo 内容时，需要保留生成记录。

用户后续查看任务时，应能知道：

- 这个任务来自哪条原始输入。
- AI 为什么这样判断。
- AI 生成了哪些字段。
- 用户是否修改过。
- 用户是否把它移出 todo。

第一阶段不要求把日志做成复杂审计系统，但必须保证产品上可追溯、可校对。

## 11. 典型场景

### 场景 A：明确 todo

用户输入：

```text
下周三前把 AI todo app 的 MVP 文档提交评审
```

系统行为：

- 保存原始输入。
- 判断为明确 todo。
- 自动创建任务。
- 任务进入 Todo 数据库。
- 用户可查看来源和 AI 生成记录。

### 场景 B：模糊 todo

用户输入：

```text
周末研究一下 Cubox、OmniFocus、Todoist
```

系统行为：

- 保存原始输入。
- 判断为模糊 todo。
- 进入待澄清区。
- AI 给出建议追问或下一步。

### 场景 C：非 todo 信息

用户输入：

```text
AI todo app 可以借鉴 Cubox 的稍后读体验
```

系统行为：

- 保存原始输入。
- 判断为非 todo。
- 进入隔离区。
- 不参与任务计划。

### 场景 D：纠正 AI 误判

用户在 Todo 数据库中看到一条并不需要行动的内容。

用户点击“移出 todo”。

系统行为：

- 该条目移出正式任务系统。
- 进入非 todo 或隔离状态。
- 保留原始输入和生成记录。

### 场景 E：用 `@` 引用任务或项目

用户输入：

```text
帮我把 @AI Todo PRD 拆成今天能做的 3 个小任务
```

系统行为：

- 用户输入 `@` 后可搜索任务或项目。
- 选中引用对象后，AI 基于该对象继续处理。
- 新生成内容继续保留 raw data 和生成记录。

## 12. 阶段路线

### v0.1 Demo

目标：验证 todo-first 的基本方向。

重点：

- 聊天输入。
- AI 分类。
- 任务、待澄清、非 todo 的基本分流。
- 初步 Web demo。

### v0.2 内部真实试用

目标：从能演示走向内部可用。

重点：

- 账号系统。
- 真实 AI API 接入。
- Codex 登录能力预研。
- Web 三段式响应式布局。
- Todo 数据库可编辑。
- raw data 与生成记录可追溯。
- AI 误分类可移出。
- 内部只读后台。

### 后续阶段

可能方向：

- 更完整的项目管理。
- 更丰富的视图和筛选。
- 多 Agent。
- 日历或外部输入集成。
- 浏览器插件。
- 移动端体验。
- 企业权限和协作。
- 更完整的数据导出和审计。

---

## 13. 非目标

当前阶段不做：

- CLI 优先体验。
- 移动原生 App。
- 商业化账号体系。
- 复杂组织权限。
- 计费系统。
- 多 Agent 编排。
- 飞书、微信、Obsidian、Cubox 等外部输入集成。
- 完整知识库。
- 完整端到端加密。

---

## 14. 成功标准

v0.2 成功的标志不是功能很多，而是内部团队愿意真实使用。

最低成功标准：

- 用户能登录并输入真实任务。
- AI 能把大部分明确 todo 自动进入数据库。
- 用户能快速修正 AI 误判。
- 用户能在 Todo 数据库里继续编辑任务。
- 用户能追溯任务来源。
- 团队能通过后台看到测试数据和问题。
- Web 在桌面和移动端都能完成核心流程。

## 15. 文档维护原则

本文档是产品基线，不替代技术设计文档。

产品侧负责明确：

- 为什么做。
- 给谁用。
- 用户如何使用。
- 哪些行为必须成立。
- 如何验收。

技术侧负责明确：

- 架构。
- 数据库。
- 接口。
- 模型调用。
- 权限实现。
- 部署和监控。
